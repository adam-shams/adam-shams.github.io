🏈 Discovery Project

CFB Play-Call Tendency Analyzer — a Streamlit web app that surfaces situational play-calling patterns for any Power-5 college football team using live data from the College Football Data API and LLM-generated scouting reports.

Project at a Glance

The CFB Play-Call Tendency Analyzer is a Streamlit application that pulls play-by-play data from the College Football Data (CFBD) API and lets the user slice an offense's tendencies across down, distance, field zone, score situation, and quarter. It generates run/pass distributions, play-type frequency charts, field-position breakdowns, and AI-written scouting summaries — turning hundreds of thousands of raw play records into something a coach, analyst, or any curious fan can actually read.

The Idea

I've gotten a lot more into college football since coming to GT and one of the things that always fascinated me is how much of the game lives below the surface — coaches script openings, lean on certain tendencies on third down, and call dramatically different plays inside their own 10 versus the red zone. That information exists in public play-by-play data, but it's locked inside hundreds of thousands of rows that nobody is going to scroll through and I wanted a clear way to visualize the patterns.

From there the goal was to build a tool that anyone could open in a browser, pick a team and a season, and get a real read on how that offense actually behaves in certain conditions. I wanted to combine clean data engineering with LLMs to help summarize raw stats into language a person can act on.

Project Progress

Scoping & data exploration. I started by exploring the College Football Data API to understand what the play endpoints actually return — fields like down, distance, yards-to-goal, play type, score, and quarter. I scoped to Power-5 teams and seasons 2021–2025 to keep the data volume reasonable.

Caching layer. Pulling a full season of plays for every interaction was painfully slow, so I added a local JSON cache keyed by team and season. After the first fetch, every subsequent filter is instant. This was one of the most satisfying improvements to make — going from a 10+ second wait to a snappy interaction.

Modular analysis engine. I broke the logic into fetch_plays.py, filter_plays.py, analyze_plays.py, and summarize.py, with a shared utils/constants.py for things like field zones (Pinned Deep, Own Territory, Midfield, Opponent Territory, Red Zone) and score-situation buckets. Keeping these layers separate made it much easier to test a piece in isolation.

Streamlit UI. The interface lets the user pick a team, a season range, and apply granular filters (down, distance to first, yards-to-goal, score differential, quarter). Output includes run/pass splits, play-type frequency, field-zone breakdowns, the most common down/distance/zone combinations, and a CSV export of the filtered plays.

LLM scouting reports. I integrated Groq (Llama-3.1-8b-instant) as the default backend with optional local Ollama support, prompting the model to produce a short, analytical 3–4 sentence scouting summary from the computed statistics.

Deployment. The app is deployed on Streamlit Community Cloud so anyone can use it without installing anything locally.

Successes & Failures

Success — caching. Adding the JSON cache was the biggest single win in the project. It transformed the experience from long wait times every time to fetching once and then explore freely, and it made the LLM scouting reports feel responsive enough to actually be useful.

Success — modular architecture. Splitting the codebase into small, single-purpose modules (fetch, filter, analyze, summarize) paid off every time I needed to add a feature or debug. I could change the field-zone definitions without touching the LLM code, or swap the LLM backend without touching the analysis layer.

Failure / lesson — context limits. My first attempt at the AI scouting report dumped the full filtered play list into the prompt. The model either ran out of context or produced vague, generic output. I had to redesign the summarization step to feed the LLM the computed statistics (percentages, averages, top combinations) instead of raw plays — a cleaner abstraction and a much better summary.

Failure / lesson — UI iteration. The first version of the Streamlit UI tried to show every filter and every chart on one screen and was overwhelming. I had to step back and reorganize sections so the app reads top-to-bottom in a logical order: pick team -> see overall splits -> drill into situations -> read the summary.

ECE Skills Gained

This project pushed me on the computing systems and information internetworks sides of ECE. I got to play around more with data analytics and visualizations as even though this wasn't my original project idea that was always the core of my inital pitch. Plus it fits with threads such as Distributed Systems + Signal and information processing:

  • API integration & data engineering: Working with a real public API (CFBD), handling pagination, and designing a caching layer that respects rate limits.
  • System decomposition: Splitting a non-trivial program into clean modules with well-defined responsibilities — the same architectural thinking that shows up in digital design and embedded systems.
  • LLM prompt engineering: Learning when to send raw data vs. pre-computed features to a model, and how to constrain output length and tone.
  • Cloud deployment: Deploying to Streamlit Community Cloud, managing secrets (API keys), and configuring the runtime.
  • UI/UX iteration: Treating the user interface as a real engineering problem — what to show, in what order, with what defaults.
  • Version control discipline: Using Git/GitHub to manage incremental changes and keep the project deployable at every step.

Final Thoughts

This project reinforced my interest in the computing-systems side of ECE. I came in already excited about LLMs and ML, but the parts I enjoyed most were the ugly parts: getting the cache right, structuring the modules, deciding what the LLM should and shouldn't see. Those are the skills that make the ML layer actually work in production.

The next thing I want to do is finish the MCP (Model Context Protocol) capabilities — exposing the analysis tools (fetch_plays, filter_plays, analyze_plays, summarize) as a proper MCP server so that Claude, Cursor, or any MCP-compatible client can call them directly. That turns the analyzer from a single Streamlit app into a reusable backend making it so you can basically "chat" with the product.

Other potential next steps are: defensive tendency analysis (right now it's offense-only), opponent-adjusted stats, multi-game trend lines, and expanding beyond Power-5. I'd also like to experiment with smaller local models via Ollama to see how close they can get to the Groq-hosted reports without the network round-trip.

More broadly, the Discovery Project format gave me space to scope, build, ship, and reflect on something I cared about in a low pressure setting and it surfaced exactly the kind of full-stack ECE work I want to do more of.
